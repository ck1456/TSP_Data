The Travelling Salesperson Challenge

Your goal is to visit each city exactly once and return home (to the first city) by travelling the shortest total distance. You will write a program that determines the order in which you travel to all of the cities. Your solution will be compared to all of the other travelling salespeople out there and the shortest total distance wins. Details for the specifcation of your program, the input data, and expected output data are as follows:

Executing your program

  • Your program must run at the command line and accept two command line arguments. For example: 
    >>travel <input-file> <output-file>
    It can be a java program, python, c++, bash script as long as you provide the command to execute it and it accepts the two required command line arguments.
  • The first command-line argument will be a relative path to an input file. This is where you will read the configuration of the cities from. You will find more detail about the input format and sample files below.
  • The second command-line argument will a relative path to an output file. This is where we expect you to write your output. The output should consist of exactly one line with a space-separated list of city ids that represent the tour you will travel.
  • Your output must contain as many numbers as there are (rows) cities in the input. Do not list the starting city again at the end. Any solution that does not contain a valid permutation of the input city ids will be disqualified.
  • Any solution that is not properly formatted according to this specification is subject to disqualification.
  • Your program must execute and terminate within 5 minutes. Any program still running after 5 minutes will be forcefully terminated. If the program has produced a valid solution at this point, it will be evaluated.
  • Unless otherwise specified, and special arrangements are made ahead of time with the architect team, all programs will be executed on energon1.

Input data format and samples

  • All input data is taken directly from the comprehensive University of Waterloo listing located here: http://www.math.uwaterloo.ca/tsp/data/usa/usa115475.tsp
  • Sample data is provided in the same format as the original data source (except with no extra lines). Each line corresponds to one city and is represented as [n] [x] [y] where n is a positive integer and x and y are floating point numbers. The three values are separated by a single ASCII space. For example: 
    14 31277.947300 86125.779600
  • Sample data is provided at data/sample_[n].tsp for 1 ≤ n ≤ 5
  • Test data will be provided at data/test_[n].tsp for 1 ≤ n ≤ 5 (but it's not available yet!)
  • Sample data is representative of the form of the test data, but not the content.

Evaluating Competing Solutions

  • The score for you solution is the total distance that your proposed tour takes. You don't have to calculate this, we will calculate this in order to make sure it is done correctly and consistently for all teams.
  • The team with the shortest tour length for a given round gets a score of 1. All other scores for that round will be expressed as a fraction of the best score. For example: Team A gets a total length of 4000, and Team B gets a total length of 4800. Team A's normalized score for the round is 1.0 and Team B's is 1.2. Any team that is disqualified for the round will receive a score of 1.5.
  • There will be 5 final test data sets.
  • The team with the lowest sum total of 5 scores wins big.

FAQ - Frequenty Asked Questions

Who is behind all of this?

Chris Keitel (ck1456@nyu.edu) and Kumar Prabhu (kumar.prabhu@nyu.edu). You can blame them for your troubles. And ask more questions if you have them.
Will the city ids be in order?
No, the city ids will not necessarily be in order. Although they might.

What is the most number of cities we have to handle?

The original data set has 115,475 cities, so it won't be more than that! In general, the samples are representative of the data sets that will be provided for the competitive evaluation.

Are the cities always numbered 1 ... n?

No, the city ids in the sample data correspond precisely to the city ids in the original full data set. In this way, each sample and test set is a proper subset of the full USA data set. This prevents ambiguity between test runs and prevents confusion all around.

Is there anyway I can make sure my code works before the competition?

Yes! Kumar will tell you how.
